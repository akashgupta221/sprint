# Sprint — Agile Project Management

Sprint is a small, opinionated agile project management tool for software teams. It models the work hierarchy that most teams actually use day-to-day:

```
Project ─┬─ User Story ─┬─ Task
         │              ├─ Task
         │              └─ Task
         └─ User Story ─┬─ Task
                        └─ Task
```

You can browse a dashboard, drill into projects, manage user stories and tasks (including assignment, status, due dates, and estimates), invite team members, and watch a real background email outbox deliver assignment / status notifications with retries and visible failure handling.

This repository was built as a take-home assignment. It is a **pnpm monorepo** with a single React + Vite frontend artifact, a single Express API artifact, and a shared OpenAPI contract that drives both sides via codegen.

## Quick start

The Replit workflows take care of this automatically, but for completeness:

```bash
# 1. install
pnpm install

# 2. push the database schema (uses DATABASE_URL)
pnpm --filter @workspace/db run push

# 3. seed sample data (4 members, 2 projects, 4 stories, 6 tasks, activity feed)
pnpm --filter @workspace/api-server run seed

# 4. run everything
pnpm dev
```

The web app is served at `/` (the root preview path) and the JSON API at `/api`. Interactive Swagger UI documentation is at `/api/docs`.

## Architecture

```
┌──────────────────────┐    JSON / OpenAPI    ┌───────────────────────┐
│   React + Vite SPA   │ ◄──────────────────► │   Express 5 API       │
│   (artifacts/        │      /api/...        │   (artifacts/         │
│    agile-pm)         │                      │    api-server)        │
└──────────┬───────────┘                      └──────────┬────────────┘
           │                                             │
           │  React Query hooks                          │  Drizzle ORM
           │  generated by Orval from openapi.yaml       │
           │                                             ▼
           │                                  ┌──────────────────────┐
           │                                  │   PostgreSQL         │
           │                                  │   members, projects, │
           │                                  │   stories, tasks,    │
           │                                  │   activity_events,   │
           │                                  │   notifications      │
           │                                  └──────────────────────┘
           │                                             ▲
           │                                             │
           │                                  ┌──────────┴───────────┐
           │                                  │ Notification worker  │
           │                                  │ (in-process, polls   │
           │                                  │  the outbox table    │
           │                                  │  with FOR UPDATE     │
           │                                  │  SKIP LOCKED)        │
           │                                  └──────────┬───────────┘
           │                                             │
           │                                             ▼
           │                                       SMTP (or JSON
           │                                       fallback in dev)
```

Key design choices:

- **OpenAPI as the single source of truth.** `lib/api-spec/openapi.yaml` describes every endpoint and schema. Orval generates two libraries from it: `@workspace/api-zod` (runtime validators used by the server) and `@workspace/api-client-react` (typed React Query hooks used by the web app). When the spec changes, both sides change together — no drift.
- **Drizzle ORM with a real Postgres.** Schema lives in `lib/db/src/schema/*.ts`. `drizzle-kit push` keeps the dev database in sync; the same schema would feed real migrations in production.
- **Express 5, thin routes.** Each route file validates input with the generated Zod schemas, performs the database work, and emits two side effects when relevant: an **activity event** (for the dashboard feed) and an **outbox notification** (for the async worker).
- **In-process background worker.** Email delivery is intentionally decoupled from the request path. Routes only enqueue rows into a `notifications` outbox table; a polling worker claims and delivers them, with exponential backoff and a dead-letter state. This pattern survives process restarts and avoids losing notifications on transient SMTP failure.

## Data model

| Table | Purpose |
| ----- | ------- |
| `members` | People on the team. `name`, `email` (unique), `role` (admin / member). |
| `projects` | Top-level container. `key` (unique short code), `name`, `description`, `status` (active / archived / completed), optional `owner_id` → members. |
| `stories` | User stories under a project. `title`, `description`, `acceptance_criteria`, `status` (backlog / ready / in_progress / review / done), `priority`, `story_points`. Cascades on project delete. |
| `tasks` | Concrete work items under a story. `title`, `description`, `status` (todo / in_progress / blocked / done), optional `assignee_id` → members, `estimate_hours`, `due_date`. Cascades on story delete. |
| `activity_events` | Append-only feed used by the dashboard. Records `kind`, human-readable `message`, and references to the related project / story / task. |
| `notifications` | Persistent email outbox. Each row tracks `kind`, `recipient_email`, `subject`, `body`, `status`, `attempts`, `max_attempts`, `last_error`, `next_attempt_at`, `sent_at`. The worker reads and updates this table. |

All tables use serial primary keys and timestamptz columns with `defaultNow()`.

## Async workflow: the email outbox

The assignment requirement is "at least one asynchronous workflow with retry and failure handling." Sprint implements this as a transactional outbox + polling worker:

1. **Enqueue (synchronous side).** When a task is created with an assignee, when an existing task's assignee changes, or when an assigned task's status changes, the route handler inserts a row into `notifications` with `status='pending'` and `next_attempt_at=NOW()`. This insert lives in the same logical transaction as the user-facing change, so we never silently drop a notification because of a crash between "save the task" and "send the email."

2. **Worker tick.** `notificationWorker.ts` runs every 5 seconds (configurable via `NOTIFICATION_POLL_MS`). On each tick it claims a batch of due rows with a single SQL statement:

   ```sql
   WITH due AS (
     SELECT id FROM notifications
     WHERE status IN ('pending','failed')
       AND next_attempt_at <= NOW()
       AND attempts < max_attempts
     ORDER BY next_attempt_at ASC
     LIMIT $batch
     FOR UPDATE SKIP LOCKED
   )
   UPDATE notifications SET status='sending', updated_at=NOW()
   WHERE id IN (SELECT id FROM due)
   RETURNING *;
   ```

   `FOR UPDATE SKIP LOCKED` makes the worker safe to scale horizontally — multiple instances can poll the same table without ever processing the same row twice.

3. **Deliver.** Each claimed row is sent via Nodemailer. If `SMTP_HOST` is set, it uses real SMTP; otherwise it falls back to Nodemailer's `jsonTransport`, which serializes the message instead of sending it. This means the entire pipeline (enqueue → claim → "deliver" → mark sent) works end-to-end with no external infrastructure, and switching to a real provider is a matter of setting four env vars.

4. **Retry with exponential backoff.** On failure, the row is moved to `status='failed'`, `attempts` is incremented, `last_error` is recorded, and `next_attempt_at` is pushed out by `30s × 2^(attempts-1)` plus jitter (~30s, 1m, 2m, 4m, 8m). The next worker tick picks it back up automatically.

5. **Dead-letter.** Once `attempts` reaches `max_attempts` (default 5), the row is moved to `status='dead_letter'` and stops being polled. The Notifications page surfaces these rows prominently and offers a manual **Retry** button (`POST /notifications/:id/retry`) which resets `attempts=0` and `status='pending'`, putting the row back into the queue.

6. **Observability.** The worker logs every send with the message ID and current transport mode; the `/notifications` page in the UI auto-refreshes every 5 seconds and shows attempts, last error, and current status, so you can watch the queue drain in real time.

This pattern is used in production by, for example, the Stripe and Shopify webhook pipelines — it gives you at-least-once delivery, durability across restarts, and bounded retry without reaching for a separate queue service.

## Configuration

The only required variable is `DATABASE_URL` (a Postgres connection string). Replit provisions one automatically.

| Variable | Default | Purpose |
| --- | --- | --- |
| `DATABASE_URL` | required | Postgres connection string. |
| `PORT` | required | Port the API server binds to. |
| `SESSION_SECRET` | provided | Reserved for future auth work. |
| `SMTP_HOST` | _unset_ | Set to enable real SMTP delivery. When unset, the worker uses Nodemailer's JSON transport so the pipeline still runs end-to-end. |
| `SMTP_PORT` | `587` | |
| `SMTP_SECURE` | `false` | Set to `true` for SMTPS. |
| `SMTP_USER`, `SMTP_PASS` | _unset_ | Optional SMTP credentials. |
| `EMAIL_FROM` | `Sprint <no-reply@sprint.local>` | From-address used by the worker. |
| `NOTIFICATION_POLL_MS` | `5000` | How often the worker polls the outbox. |
| `NOTIFICATION_BATCH_SIZE` | `10` | Max rows claimed per tick. |

## Testing the app manually

1. Open the web app at `/`. The dashboard shows seeded counts and a recent activity feed.
2. Navigate to **Projects** → open `Customer Web App` → open a story → re-assign a task to a different member, or change its status.
3. Navigate to **Notifications**. Within ~5 seconds you'll see a new row appear with `status='sent'` (or `failed` and then a retry).
4. To see the failure path, set `SMTP_HOST=invalid.example.com` in env vars and restart the API. New notifications will move from `pending` → `failed` → `failed` → `dead_letter` over a few minutes, with the error visible in the table. Click **Retry** to put them back in the queue.

## Security notes

- Input is validated server-side against generated Zod schemas; the same schemas are shipped to the client for symmetric type safety, but the server never trusts client-side validation.
- Database access uses parameterised queries via Drizzle — no string concatenation into SQL.
- Cascade rules are explicit on foreign keys (delete a project → its stories and tasks go with it; delete a member → tasks lose their assignee but are preserved).
- Outbox emails are stored on the server only and never returned to the client unencrypted beyond the audit fields shown in the UI.
- Authentication is provided by **Clerk Auth**. The web app routes `/`, `/sign-in`, `/sign-up`, and `/api/healthz` + `/api/docs` are public; everything else requires a signed-in session.
- The Express API is gated by `clerkMiddleware()` plus a `requireAuth` handler — unauthenticated requests to `/api/projects`, `/api/dashboard/*`, etc. return `401 Unauthorized`.
- CORS is open in development; in production it should be locked to the deployed origin.

## What I'd do next

If this were going to production I would, roughly in order:

1. **Tie Clerk users to workspace members.** Today every authenticated user sees the same shared workspace; the next step is provisioning a `members` row on first sign-in (storing the Clerk user ID), so `assigneeId` and notification recipients can resolve to the actual signed-in user instead of the seeded fixtures. This also unlocks per-project membership / role enforcement.
2. **Drizzle migrations instead of `push`.** Generate proper migration files and run them on deploy.
3. **Move the worker out of the API process.** A separate `worker` artifact polling the same outbox would let me scale and restart the API independently.
4. **Real-time updates.** Push activity and notification updates over Server-Sent Events instead of polling, so the UI feels live without refresh churn.
5. **Optimistic UI** for status changes on the Kanban board and stronger empty / error states in places where the assistant's first pass was thin.
6. **Tests.** Vitest for the route handlers (using a throwaway Postgres schema) and a couple of Playwright happy-path tests for the critical flows: create project → add story → add task → reassign → see notification deliver.
7. **Permissions on notifications.** Today anyone with API access can read every email subject. In production these would be gated to the original recipient + project admins.
